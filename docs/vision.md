# DevPane — ビジョンと設計方針

## コンセプト

**寝てる間に勝手に動くAI開発チーム。起きたらDiscordに進捗報告が来ている。**

人間の役割は「方針を決める」と「報告を見て一言返す」だけ。
タスク生成・テスト設計・実装・レビュー・PR管理はすべて自律で回る。

## 人間のワークフロー

```
朝 or 帰宅後:
  1. DiscordでPR Agent日報を見る
     | # | タスク        | 変更      | テスト | リスク | 判定     |
     |---|--------------|-----------|--------|--------|---------|
     | 1 | チャット機能   | +120/-3   | 全通過 | 低     | ✅推奨  |
     | 2 | DB移行        | +45/-200  | 1失敗  | 高     | ⚠️要確認 |
     | 3 | README更新    | +5/-2     | -      | 無     | ✅推奨  |

  2. 「1,3マージ、2クローズ」と返信 ← これだけ

  3. 方針変更があればCLAUDE.md編集 or チャットで一言
     「テスト周り強化して」→ PMの記憶に入る

  4. 閉じて放置 → またAIが回り始める
```

**人間はdiffを読まない。** PR Agentが安全性を判定して推奨を出す。
人間はパッと見でおかしい箇所があれば番号指定でクローズするだけ。

## 開発パイプライン

### 全体フロー

```
┌─────────────────────────────────────────────────────────┐
│ 開発ループ（24h自動）                                      │
│                                                         │
│  PM ──→ 監査 ──→ テスター ──→ 監査 ──→ Worker ──→ PR作成  │
│  仕様    方針     テスト      仕様と     実装     ブランチ   │
│  生成    チェック  作成       テスト照合   (TDD)   push     │
│                                                         │
│  → 記憶更新 → 次サイクルへ                                 │
└─────────────────────────────────────────────────────────┘
         ↓ PRが溜まる
┌─────────────────────────────────────────────────────────┐
│ PR Agent（日次バッチ）                                     │
│                                                         │
│  未マージPR収集 → diff要約 → テスト結果 → リスク評価         │
│  → Discordにテーブル形式で投稿                              │
│  → 人間が「1,3マージ、2クローズ」と返信                      │
│  → マージ/クローズ実行 → 結果を記憶にフィードバック           │
└─────────────────────────────────────────────────────────┘
```

### パイプライン詳細

各ステップは明確な入出力を持ち、どこで止まっても理由がわかる。

#### Step 1: PM — 仕様生成
```
入力: CLAUDE.md + 記憶 + タスク履歴
出力: タスク仕様（タイトル、説明、受け入れ基準）
許可ツール: Read, Glob, Grep（読み取り専用）
```

PMは方針をタスクに分解する。実装の詳細は書かない。
受け入れ基準（acceptance criteria）を明示し、テスト設計の入力にする。

#### Step 2: 監査 — 方針チェック
```
入力: 生成されたタスク仕様 + 記憶 + 方針
出力: 判定（pass/fail + 理由）
許可ツール: Read, Glob, Grep（読み取り専用）
```

- タスクが方針（CLAUDE.md）と矛盾していないか
- 既存featureの破壊を含んでいないか
- 重複タスクがないか

ルールベースでは拾えない曖昧な矛盾を判定する。
fail時はタスクを却下し、理由を記憶に記録。

#### Step 3: テスター — テスト作成
```
入力: タスク仕様（受け入れ基準）
出力: テストファイル
許可ツール: Read, Edit, Write, Glob, Grep
```

仕様の受け入れ基準からテストコードを先に書く。
実装より先にテストが存在する = TDDの自動化。

#### Step 4: 監査 — 仕様とテストの照合
```
入力: タスク仕様 + テストファイル
出力: 判定（カバレッジ十分か、仕様の漏れはないか）
許可ツール: Read, Glob, Grep（読み取り専用）
```

テストが仕様を網羅しているか確認。漏れがあればテスターに差し戻し。

#### Step 5: Worker — 実装（TDD）
```
入力: テストファイル + タスク仕様
出力: 実装コード（テストが通る状態）
許可ツール: Read, Edit, Write, Bash, Glob, Grep
```

テストが既にある状態で実装する。テスト全通過 = 仕様を満たしている。
worktreeで隔離実行。mainを直接触らない。

#### Step 6: PR作成
```
入力: worktreeのコミット + Observable Facts
出力: GitHub PR
```

gh pr createでブランチをpush。PRのbodyに変更概要、diff stats、テスト結果を記載。

#### Step 7: PR Agent — 日次レビュー（ループ外）
```
入力: 未マージPR一覧 + 各PRのdiff + テスト結果
出力: Discordへの日報テーブル
許可ツール: Read, Glob, Grep（読み取り専用）
```

開発ループとは独立して日次で実行。
各PRの安全性を評価し、マージ推奨/要確認/非推奨を判定。
人間の返信を受けてマージ/クローズを実行し、結果を記憶にフィードバック。

- PRマージ → `remember('decision', '...')`
- PRクローズ → `remember('lesson', 'この方向は不要と判断された')`

### 役割の実装

全役割が同じClaude CLI呼び出しの引数違いで実装される。
エージェント数は増えない。schedulerが順番に呼ぶだけ。

| 役割 | 実体 | 許可ツール | コスト |
|------|------|-----------|--------|
| PM | `claude -p` | Read,Glob,Grep | 軽量 |
| 監査 | `claude -p` | Read,Glob,Grep | 軽量 |
| テスター | `claude -p` (worktree内) | Read,Edit,Write,Glob,Grep | 中量 |
| Worker | `claude -p` (worktree内) | Read,Edit,Write,Bash,Glob,Grep | 重量 |
| PR Agent | `claude -p` + `gh` | Read,Glob,Grep | 軽量（日次） |

常駐エージェントはゼロ。全てワンショット呼び出し。

## 人間とAIの役割分担

```
人間（経営者）
  └─ 方針をPMに投げる。日報を見て一言返す。たまにWeb UIで覗く。

PM（プロジェクトマネージャー）
  └─ 方針→タスク分解、仕様作成、記憶管理

監査（品質管理）
  └─ 方針チェック、仕様-テスト照合、矛盾検出

テスター（QA）
  └─ 仕様からテスト作成

Worker（実装者）
  └─ テストを通す実装

PR Agent（報告役）
  └─ 日次PR要約、人間への報告、マージ/クローズ実行
```

人間はPMの上に立ち、方針と意見を投げて調整する奥役。
細かいPRレビューや実装判断には関わらない。

## 安全装置

| 層 | 仕組み | 目的 |
|----|--------|------|
| worktree隔離 | mainを直接触らない | コードの安全性 |
| 監査ステップ | 方針逸脱・矛盾を検出 | タスク品質の担保 |
| TDD | テスト先行で実装 | 実装品質の担保 |
| PR出力 | 自動マージしない | 成果物の追跡性 |
| PR Agent判定 | リスク評価付き日報 | 人間の判断コスト削減 |
| 記憶層 | SQLiteにfeature/decision/lesson | 「作って壊す」防止 |
| Observable Facts | 客観指標で完了判定 | AI自己申告に依存しない |
| レートリミットbackoff | 指数バックオフ | API制限への対応 |

## 記憶層

### テーブル

```sql
CREATE TABLE memories (
  id             TEXT PRIMARY KEY,
  category       TEXT NOT NULL,  -- 'feature' | 'decision' | 'lesson'
  content        TEXT NOT NULL,
  source_task_id TEXT,
  created_at     TEXT NOT NULL,
  updated_at     TEXT NOT NULL
);
```

### 書き込みルール

| トリガー | category | 書き込み元 |
|---------|----------|-----------|
| タスク成功、新規ファイル | feature | ルールベース（機械的） |
| ファイル削除 | - | forget（機械的） |
| PRマージ | decision | PR Agent |
| PRクローズ | lesson | PR Agent |
| 人間のチャット指示 | decision/lesson | Web UI → 記憶API |
| 監査のタスク却下理由 | lesson | 監査ステップ |

将来拡張:
- sqlite-vecによるセマンティック検索
- 記憶の自動要約・圧縮（件数が増えた場合）

## Web UI — 監視と介入

Web UIは「オフィスの窓」。チームが働いてるのを覗いて、おかしかったら介入する。

### 監視（見る）

| 画面 | 見えるもの | 判断材料 |
|------|-----------|---------|
| ダッシュボード | タスク一覧、ステータス、コスト | 全体の健康状態 |
| タスク詳細 | Workerのリアルタイムログ、diff | 変な方向に行ってないか |
| パイプライン | 各ステップの進行状況 | どこで詰まっているか |
| 記憶一覧 | PMが覚えている機能・判断・教訓 | PMの認識がズレてないか |

### 介入（触る）

| 操作 | 効果 | ユースケース |
|------|------|-------------|
| スケジューラ一時停止 | 新規タスク実行を止める | 暴走時の緊急ブレーキ |
| タスクキャンセル | 実行中/pendingを中止 | 不要なタスクを止める |
| 記憶の編集/追加 | PMの認識を直接修正 | 「この機能は触るな」等 |
| タスク手動追加 | キューにタスク投入 | 特定の修正を優先させたい |
| チャット指示 | 記憶に書き込み | 軽い方針調整 |

### 段階的な介入レベル

```
レベル0: 放置（正常系）
  → Discordの日報だけ見る。番号でマージ指示。

レベル1: 方針調整
  → チャットで一言 or CLAUDE.md編集。次サイクルで反映。

レベル2: タスク介入
  → Web UIから不要タスクをキャンセル、手動タスクを追加。

レベル3: 緊急停止
  → スケジューラ一時停止。状況を確認して再開。

レベル4: 記憶修正
  → PMの記憶を直接編集。「Xは実装済み」「Yは不要」を明示。
```

## 原則

1. **人間は経営者。AIがチーム。** 人間は方針を決めて日報を見る。タスク分解も実装もレビューもAIがやる。
2. **24時間止まらない。** 人間が寝ていても開発は進む。人間の応答を待たない。
3. **判断リソースを最小化。** 日報のテーブルを見て番号を返すだけ。diffは読まない。
4. **品質はパイプラインで担保。** 監査→テスト→実装→PR。単一のエージェントに全権限を持たせない。
5. **記憶は事実ベース。** LLMの気分で記憶が変わらないよう、トリガーとルールで管理する。
6. **仕組みで役割を作る。** エージェントを増やさず、同じCLIの呼び出しパターンで役割を分ける。
