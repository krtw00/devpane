# 目的・解決する課題

> Status: Active
> 最終更新: 2026-03-15

本ドキュメントは、DevPaneの目的と解決する課題を明確にする。

---

## プロジェクトの目的

DevPaneの目的は、一度起動したら放置で自律開発し続ける常駐デーモンを実現することである。人間の役割は「方針を決める」と「日報を見て番号を返す」だけに限定する。

---

## 解決する課題

### 課題一覧

| # | 課題 | 影響 | 優先度 |
|---|------|------|--------|
| 1 | AIツールは人間のトリガーが必要 | 寝ている間・外出中は開発が止まる | High |
| 2 | AIの出力品質が不安定 | 壊れたコードがmainに入る可能性がある | High |
| 3 | 人間のレビューコストが高い | diff全行を読む必要があり判断に時間がかかる | Medium |
| 4 | 同じ失敗を繰り返す | プロセスが学習しないため失敗パターンが再発する | Medium |

### 課題1: 人間のトリガー依存

| 項目 | 内容 |
|------|------|
| 現状 | AIコーディングツールはタスクごとに人間が指示を出す |
| 問題点 | 開発速度が人間の稼働時間に制約される |
| 解決策 | 常駐デーモンがタスクキューを自律消化する。キュー空時はPMが自動でタスク生成する |

### 課題2: AI出力の品質不安定

| 項目 | 内容 |
|------|------|
| 現状 | AIが生成したコードの品質は保証されない |
| 問題点 | テスト未通過・既存機能破壊のリスクがある |
| 解決策 | Observable Factsによる客観判定と3段Gate審査で品質を段階的に担保する |

### 課題3: レビューコスト

| 項目 | 内容 |
|------|------|
| 現状 | PRのdiffを人間が全行レビューする |
| 問題点 | 判断に時間がかかり、放置の妨げになる |
| 解決策 | PR Agentがリスク評価付き日報を出す。人間は番号指定でマージ/クローズするだけ |

### 課題4: 失敗パターンの再発

| 項目 | 内容 |
|------|------|
| 現状 | タスク失敗の原因分析は手動 |
| 問題点 | 同じroot causeの失敗が繰り返される |
| 解決策 | なぜなぜ分析Agentが失敗パターンを自動分析し、Gate・PMテンプレートを改善する |

---

## 成功基準

| 基準 | 指標 | 目標値 |
|------|------|--------|
| 自律稼働 | 人間介入なしでタスクが完了する | 3タスク以上/日 |
| 品質 | Gate 3通過後のPRがマージ可能 | 80%以上 |
| 放置耐性 | daemon起動後に人間が操作しない時間 | 8時間以上 |
| 自己改善 | なぜなぜ分析による失敗率の低下 | 改善前比20%減 |

---

## 非目標

| 非目標 | 理由 |
|--------|------|
| チーム利用 | まず個人で安定稼働させる。チーム対応はPhase 3以降 |
| マルチリポジトリ対応 | 単一リポジトリ（DevPane自身）で動作検証を優先する |
| GUI操作中心のUX | Web UIは監視用。主な介入はDiscordまたはCLAUDE.md編集 |
| API従量課金 | Max plan定額サブスクリプション内で運用する |
