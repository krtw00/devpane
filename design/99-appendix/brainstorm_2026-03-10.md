# DevPane — ブレスト整理 (2026-03-10)

## 1. DevPaneとは

**AIチームのバーチャルオフィスを、ブラウザの窓から覗く。**

階層型マルチエージェント（PM→リード→メンバー）が常駐で自律開発する。
人間はブラウザを開いてログを眺め、気になったらチャットで茶々を入れるだけ。
閉じても裏で回り続ける。安全だから放置できる。

### 名前の由来
- Pane = 窓ガラス。「開発の窓」からAIチームの仕事を覗く体験
- Pane = tmux pane。Shogun（マルチエージェント先行プロジェクト）のtmux並列実行からの系譜

### キーワード
- 常駐デーモン型（人間がトリガーを引かない。タスクキューを自律消化し続ける）
- 例外処理型（正常系で人間を挟まない。異常時だけ通知）
- 「監視ダッシュボード」ではなく「オフィスの窓」
- Web UI ＋ Slack/Discord通知 ＋ チャット介入

## 2. AgentMineからの継承

DevPaneの前身はAgentMine。元々Redmineの「AIタスク管理ツール」として生まれ、AI非依存の開発タスクオーケストレーション基盤へ進化した。コンセプト（自動ループ・タスク消化・報告のみ）は一貫している。名前とスコープをリセットして再出発する。

### 継承する資産（安全性・品質レイヤー）

常駐で放置するからこそ必要な安全装置であり、これらは本質機能。肥大化ではない。

- **Proof-Carrying Run**: 証跡パック自動生成（prompt hash、scope、変更ファイル、DoD結果）
- **Observable Facts**: AIの自己申告に依存せず、exit code・diff・テスト結果で状態を自動判定
- **Memory Governance**: 記憶の信頼度・有効期限・承認で長期運用の品質維持
- **スコープ制御**: write_scope + exclude でファイル単位のアクセス制御
- **worktree物理隔離**: タスクごとにブランチ+worktreeを作り衝突を物理的に防ぐ
- **デーモンアーキテクチャ**: 常駐型で自律的にタスクを拾って動く設計思想

### AgentMineで手放すもの

- 「AgentMine」の名前（Redmine由来、コンセプト進化に合わない）
- 承認型に見えるWeb UI設計（「オフィスの窓」に再設計）
- チーム対応（Phase 3）→ まず個人で回す
- 重厚なモノレポ構成（Next.js + Hono + Drizzle） → 必要に応じて再検討

### AgentMineが行き詰まった原因

設計思想・安全性レイヤーは正しかった。問題は実装面。

1. **並列ループの未完成**: タスク単発実行はできたが、並列自律ループのオーケストレーションが実装できず、普通にClaude Codeを叩くのと変わらない体験で止まった
2. **市場の追い上げ**: Claude Code Agent Teams、Zeroshot、Ralph Loop、Shogun等が登場し、並列オーケストレーション部分を吸収し始めた
3. **動く体験より前に設計が膨らんだ**: Web UI、監視ダッシュボード等、「まず自分が使いたい体験」に到達する前にスコープが広がった

## 3. Shogunとの関係

### multi-agent-shogun（yohey-w）
tmux上で動く階層型マルチエージェントシステム。将軍→家老→足軽（8体）の指揮系統でClaude Codeを並列実行。YAMLファイル通信＋send-keysでAPI呼び出しゼロの調整。dashboard.mdに状況集約。Star 1k。

**Shogunから取り入れるもの:**
- 階層型マルチエージェントの並列実行アーキテクチャ
- YAML通信（イベント駆動、ポーリングなし、APIコストゼロ）
- dashboard集約パターン
- Memory MCP（セッション跨ぎの記憶）

**DevPaneが変えるもの:**
- tmux → **Web UIで見る**（オフィスの窓体験）
- 武士ロールプレイ → **一般的なPM/リード/メンバー表現**
- tmux直接入力 → **ブラウザからチャットで茶々**
- ntfy通知 → **Slack/Discord通知**
- 安全性/証跡なし → **AgentMineの安全性レイヤーを統合**

### DevPaneの位置づけ = Shogunのアーキテクチャ ＋ AgentMineの安全性 ＋ Web UI

| レイヤー | Shogunから | AgentMineから | DevPane新規 |
|---------|-----------|--------------|------------|
| エージェント階層 | ○ | - | 一般表現に変更 |
| 並列実行 | ○（tmux + YAML） | - | - |
| 調整プロトコル | ○（YAML + send-keys） | - | - |
| サンドボックス | △ | ○（worktree隔離） | - |
| 完了判定 | △ | ○（Observable Facts） | - |
| 証跡 | × | ○（Proof-Carrying Run） | - |
| 記憶管理 | △（Memory MCP） | ○（Memory Governance） | - |
| スコープ制御 | × | ○（write_scope） | - |
| AI非依存 | △（Claude中心） | ○（RunnerAdapter） | - |
| UI | tmux | - | **オフィスの窓型Web UI** |
| 通知 | ntfy | - | **Slack/Discord** |
| 介入 | tmux直接入力 | - | **Webチャットで茶々** |
